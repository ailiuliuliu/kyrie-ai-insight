产品形态分类与格局

Agent不是一个东西，是一个三层结构 + 两种分类 + 两条路线的复杂生态

2.1 技术架构：核 + 壳 + 框架层

框架层
Runtime · Orchestrator · Dev Framework · Observability
壳层
Planning · Tools · Memory · Action
核 (LLM)
推理 · 理解 · 生成
⚡ 关键洞察：模型之间"核"的差距在缩小，"壳"和"框架层"的价值被严重低估。Dify/OpenClaw/LangGraph 的核心价值都在框架层。

目前真正商业化成功的 Agent 大多是广义 Agent——本质是 AI 增强的自动化工作流。

2.3 Anthropic 5种工作流模式

1
Prompt Chaining
A→B→C串行，每步可检查。适合复杂任务拆解。
2
Routing
输入分类→分流不同路径。适合多场景客服/多模型调度。
3
Parallelization
多路并行+聚合。适合需要多视角评估的任务。
4
Orchestrator-Workers
中央LLM分配子任务。适合不确定子任务数量的场景。
5
Evaluator-Optimizer
生成→评估→改进循环。适合需要反复打磨的输出。
Anthropic自己说："不要为了用Agent而用Agent，workflow够用就workflow。" 前3种本质是工作流，只有4和5有真正的自主决策成分。

2.4 Hugging Face Agent 6等级

L1简单处理器单步调用，无记忆
L2路由器分类+路由到不同工具
L3工具调用选择和调用多个工具
L4多步代理规划多步执行序列
L5多代理多Agent协作
L6Code Agent ⭐直接写代码执行，最高等级

2.5 两条路线之争

🏭 模型厂商路线
Model as Agent
代表：OpenAI · Google
端到端RL，模型原生具备规划/工具/反思能力
"更强模型 = 更好Agent"
VS
🦞 独立Agent路线
Context Engineering
代表：Manus · OpenClaw · Dify
MCP协议+RAG+记忆系统+Sandbox，构建Agent运行环境
"Less structure, more intelligence"
Anthropic的特殊位置：顶级模型厂商 + MCP协议发明者 + Claude Code独立应用——是两条路线的交叉点。
终局：短期Context Engineering更务实，中期两条路线融合，MCP是接口层。

三个核心洞察

从这场大讨论中提炼出的可行动判断

洞察一
Agent的真正瓶颈是"信任基础设施"，不是模型能力

88%的Pilot失败不是因为模型不够好，是因为企业不敢把业务交给一个无法评估、无法审计、成本不可控的系统。

Eval体系、Observability、Cost-per-task才是真正的稀缺基础设施。OpenClaw的"弃养潮"是这个问题的C端版本——大多数人没跑进生产，和企业的88%失败率本质一样。

→ 可行动方向：谁能提供"让Agent可信赖"的工具，谁就掌握了真正的机会
洞察二
Code Agent是Agent商业化的第一个真实突破口

Cursor $20亿、Claude Code $25亿——都是Code Agent或接近Code Agent。三个原因：

  • 程序员能评估代码质量（解决了"信任基础设施"问题）
  • Code Agent的"完成"定义清晰（代码能跑/测试通过）
  • HF把Code Agent列为L6最高等级，验证了技术逻辑
→ 可行动方向：所有想在Agent赛道找突破口的团队，先问——我的目标场景里，"完成"能不能被清晰定义？
洞察三
终局是"强模型 + Context Engineering"的融合，MCP是接口层

两条路线（Model as Agent vs Context Engineering）不是非此即彼：

  • 短期：Context Engineering更务实，模型还不够强
  • 中期：模型能力提升，Context Engineering的"补丁"价值部分被吸收
  • 终局：强模型提供推理能力 + Context Engineering提供可控性/可观测性

MCP协议是这个融合的接口层——OpenClaw、Dify、Claude Code都已支持。Anthropic同时布局两侧（强模型+MCP制定者），是最大受益者之一。

→ 可行动方向：押注MCP生态，押注"跨模型、跨云、可观测"的框架层产品
