背景

字节 Seed 数据团队的核心职责是：为基座大模型的每一次迭代提供"燃料"。这个岗位夹在研究员和标注团队/供应商之间，核心能力是把模糊的技术需求翻译成可执行的采集标准，并保证数据最终能转化为模型能力提升。

当前最核心的业务痛点：高质量过程性数据极度稀缺——公开互联网里存在的是"知识介绍"，而非"专家在真实工程环境中一步步完成复杂任务的推理过程"。

1
大模型训练三阶段认知
Pre-train → Mid-train → Post-train，每段的目的和数据需求完全不同
📐 三段训练全览
阶段目的数据形态数据量级改变的是
Pre-train 预训练建立语言理解和世界知识骨架海量互联网文本，几乎不筛选万亿 token模型"知道什么"的基础边界
Mid-train 中间训练在通用能力基础上强化特定领域高质量垂直领域文档，精心筛选千亿 token特定领域推理深度和能力天花板
Post-train 后训练改变行为模式，变成有用的助手指令-回复对、人类偏好排序十亿 token 级模型"怎么说话"和"愿不愿意做"
最关键的判断：Post-train 补不了 Pre-train/Mid-train 的能力缺口。数据质量决定模型能力天花板，后训练只是"解锁"已有能力，而不是创造新能力。DeepSeek 的启示：Pre-train 做得足够好，Post-train 可以很"省"；反过来不成立。
🔍 Mid-train 详解：什么时候必须做

判断标准只有一条：公开互联网里这类高质量数据稀不稀缺？

❌ 不需要 Mid-train
  • 日常对话、写作、翻译：互联网数据海量且质量够
  • 简单问答：公开数据覆盖充分
  • Pre-train 数据已经充足覆盖的通用场景
✅ 必须做 Mid-train
  • 数学深度推理：过程性数据稀缺
  • 医学临床诊断：推理链路数据少
  • 法律推理：判决文书多但推理链稀缺
  • AI4S（尖端科研）：极度稀缺，不做基本废
  • 金融建模决策链：几乎没有公开数据

Mid-train 和 Post-train 的本质区别：Mid-train 还是语言建模（预测下一个 token），改变的是"模型知道什么、推理有多深"；Post-train 用 SFT + RLHF，改变的是"模型怎么说话、愿不愿意做某事"。前者扩充能力，后者改变行为。

5TB STEM 数据 Sourcing 五条路线
配比建议、操作细节、注意事项一网打尽
📊 五条路线总览与配比
路线建议体量预算占比核心价值启动时机
公开学术/代码库清洗2.5TB（50%）10%覆盖基础量，成本最低第 1 周并行推进
专业数据库授权1TB（20%）40%高质量全文学术内容第 1 周开始谈，3 月交付
高校/专家定制生产200GB（4%）30%唯一来源的专家过程性数据第 2 周开始，持续产出
合成数据增强800GB（16%）10%放大种子数据，填补覆盖盲区有种子数据后开始
供应商定制采集500GB（10%）10%补充优先级场景缺口补充阶段
核心原则：用预算的 80% 搞清楚数据来源和质量，用剩下 20% 做合成扩充。5TB 的量不难凑，真正难的是"过程性"和"高质量"两个约束同时满足。
📚 路线一：公开学术/代码库——分层采集策略

核心洞察：不是爬下来就能用，关键在分层筛选。以 arXiv 为例，只提取方法论和讨论章节（而非整篇），有效数据密度提升 3–5 倍。

数据源领域采集重点
arXiv数学/物理/CS/生物/材料只提取方法论和讨论章节，解析 .tex 源码而非 PDF
S2ORC（Semantic Scholar）全学科已预处理好的结构化论文语料，直接可用
PubMed Central生物医学区分 Open Access 和仅摘要开放，只采全文
GitHub 顶级仓库（Star>1000）代码/科学计算抓 commit history + issue + PR 完整推理轨迹
Lean/Mathlib形式化数学每条都是机器可验证证明，质量极高，约 10 万条
Stack Exchange 数学/物理版专家问答过滤 score > 10 的回答，推导过程完整

被严重低估的数据类型：GitHub commit history。从 bug 出现、定位、修复到验证的完整过程，本质上就是"专家在真实工程约束下完成多步任务的过程性数据"，精准对应业务痛点。

🎓 路线三：高校/专家定制生产——三种方式

这是获取"专家思维过程"最直接的路——公开数据库和出版商都没有的，只能靠人产生。量少但价值密度最高，建议把预算的 30% 集中在这里。

A
结构化任务采集（主力，推荐）
设计模拟真实工程场景的任务，采集完整推理链路，不是答案。示例（材料科学）：给定导电率/熔点/成本约束，设计合金配方，写出完整推理过程和排除其他方案的理由。任务必须模拟真实工程约束，不能是选择题或简答题。
B
Think-Aloud Protocol（精品）
心理学经典方法，请专家边操作边说出思维过程，录音转文字。成本高但质量最高，适合采集 50–200 条精品样本，用作质检基准和种子数据。
C
结构化标注已有论文
给专家看论文，让他把方法论章节的每一个关键步骤打标签：假设建立 / 文献支撑 / 实验设计 / 异常处理 / 结论推断。适合批量生产结构化推理数据。
🤖 路线四：合成数据增强——边界与工程流程
✅ 合成数据能做
  • 放大种子数据（1000 条真实 → 合成 1 万条变体）
  • 填补子领域覆盖盲区
  • 生成负样本（错误推理 + 错误标注，训练纠错能力）
❌ 合成数据不能做
  • 替代真实专家原始推理（继承原始模型知识边界）
  • 用于 Benchmark 校准（造成循环依赖）
  • 合成:真实 比例超过 3:1
工程流程
1
选取 200–500 条高质量真实专家数据作为种子
↓
2
让 GPT-4/Claude 按种子数据的格式和风格生成新任务
↓
3
自动过滤：去重 / 格式检查 / 基本逻辑验证
↓
4
专家抽检 5–10%，不合格整批返工
↓
5
合格数据混入真实数据，合成:真实 ≤ 3:1，追踪模型 loss 变化

模型坍塌的早期信号：生成内容多样性下降（重复表达增多）/ 真实 Benchmark 表现下降 / 推理链路越来越短或模板化。出现以上信号立即停止。

📋 路线五：供应商 SOW 六要素

SOW（工作说明书）是决定数据质量的核心文档，写得不好会拿到一堆垃圾数据。

1
任务定义
子领域 + 难度等级（本科/研究生/研究前沿）+ 任务格式（开放推理 vs 选择题 vs 证明题）。不能只写"数学推理数据"。
2
标注者资质
相关领域硕士及以上，或 3 年以上行业从业经验；设置 qualification test，低于 80 分不录用；要求提供标注者 credential 样本，签约前审核。
3
推理链路深度
最少步骤数（根据难度定 5–15 步）；关键节点标注（假设建立/关键转折/结论验证）；禁止跳步——不能从问题直接跳到结论。
4
质量检验机制
双盲标注（同一条数据两人独立完成），一致率 > 85%；甲方保留抽检权，不合格率 > 10% 整批返工。
5
数据安全
禁止供应商用于自身模型训练；标注者不得保留数据副本；签 NDA。
6
交付格式
统一 jsonl，每条含：原始问题 / 完整推理链路 / 最终答案 / 难度标注 / 子领域标注 / 标注者 ID（用于追责）。
3
五层数据质量验证体系
从人工抽检到模型探针，确保每分钱买的数据都真正有用
🔬 第 1 层：领域专家抽检（分层抽样）

不能随机抽检，要分层抽样：每个子领域至少 1% / 供应商新加入标注者的数据抽 5% / 极端长短数据各抽 3%。

评分维度含义
领域准确性推理过程有没有领域性错误
步骤完整性有没有跳步，每步前提是否明确
语言清晰度表达是否清晰准确
推理规范性是否符合该领域的规范表达方式
难度准确性打的难度标签和实际难度是否匹配
⚙️ 第 2 层：可执行性验证（机器自动化）
代码数据（100% 自动）
  • 搭建 Docker 沙箱执行环境
  • 每条代码直接执行，记录运行结果和边界条件
  • 执行失败的数据直接打标，不进训练
数学数据（部分自动）
  • 有数值结果的题目用 Python + SymPy 验证最终答案
  • 验证通过率 < 90% 的批次需要返工
  • 形式化数学（Lean）可 100% 机器验证
🧪 第 3 层：模型探针评估（最终黑盒验证）

最有说服力的验证方式，直接测数据对模型是否有提升价值。

1
随机抽取 5,000–10,000 条买来的数据
↓
2
在基座模型上 SFT（仅用这批数据）
↓
3
在目标 Benchmark 上评测：MATH / GSM8K / MMLU-STEM 子集
↓
4
对比 baseline，提升幅度 > 2% 认为数据有效；否则停下来和供应商复盘
📊 第 4 层：数据分布分析
D
子领域覆盖
用 topic model 或 TF-IDF 聚类，检查分布均衡性——不能 80% 都是同一个子领域
D
难度分布
频率分布应接近正态，不能全堆简单或困难两端
D
推理链路长度分布
极端短（< 3 步）和极端长（> 50 步）的数据比例不超过 5%
🛡️ 第 5 层：Benchmark 污染检测

工具：MinHash LSH，计算与公开 Benchmark 数据集的 Jaccard 相似度，字符级 n-gram 相似度 > 50% 直接剔除。

必须覆盖的 Benchmark 列表：MATH / MMLU（STEM 子集）/ SciQ / ARC / GPQA / AIME / AMC

这个问题比想象中普遍——如果训练数据里包含了评测集题目，模型 Benchmark 表现会虚高，但实际能力并没有那么强。

岗位实际工作方式
你的工作界面、需求管理、价值度量、沟通策略
🔄 工作界面：你夹在谁和谁之间
研究员 ──[提需求：模糊的技术判断]──▶ 数据 PM（你） │ [翻译成可执行标准] │ ▼ 标注团队 / 外部供应商 │ [交付数据] │ 数据 PM（你） ◀──[质检/清洗/格式化/打回]──────┘ │ │ [入库 + 追踪模型提升] ▼ 模型团队 ──[反馈：这批数据有没有用]──▶ 数据 PM（你）（下一轮迭代）

核心技能：语言翻译。研究员说的"推理链路质量好"是模糊的技术判断，你要能把它拆解成标注者能执行的具体标准——几步、什么格式、如何验证、怎么算合格。翻译得好不好，直接决定数据能不能被用上。

📈 衡量工作价值的指标体系
过程指标（你能控制）
  • 数据采集完成率（计划 vs 实际）
  • 数据质检通过率
  • 数据交付周期（需求提出到可用）
  • 标注一致率（衡量 Guideline 质量）
结果指标（和研究员对齐）
  • 每批数据的"探针提升"（1 万条 SFT 测提升）
  • 数据入库率（采了多少被实际训练用）
  • Benchmark 提升中可归因于数据的比例
💬 和研究员沟通：三大摩擦点处理
常见摩擦点处理方式
需求过于宽泛（"采些高质量理科数据"）主动拆解，给出具体子任务清单，让对方确认后再开始执行
质量判断是直觉的（"感觉这批数据不好"）把直觉转化为可量化指标，建立标准，避免反复返工
需求优先级随时变化书面需求记录 + 变更管理机制，不靠口头约定
🏗️ 数据基建产品化：从项目制到持续积累

初级做法是"项目制"——研究员要了才采。成熟做法是"产品化"：

P
数据资产管理
每批数据有完整 metadata：来源 / 质量分 / 使用历史 / 对模型提升的实测效果
P
持续采集储备
核心领域数据持续积累，不等需求来了再开始——类比备货而非按单生产
P
版本管理
历史数据可追溯、可复现（类比代码的 Git），避免"数据债"
P
供应商质量追踪
追踪每家供应商的历史质量分，提前发现质量下滑趋势，干预而不是被动接受
5
行业生态与生涯认知
这个岗位的战略地位、信号判断和长期价值
🗺️ 产业链坐标
模型公司（数据消费方）：Seed / OpenAI / 智元 / Figure AI │ 数据需求 ▼ 内部数据团队（字节 Seed AI 数据&安全团队） → 自给自足，不对外服务 → 核心 KPI：模型迭代速度，而非收入 │ │（同一层级） ▼ 外部独立数据服务商：Scale AI / 光轮智能 / 觅蜂 → 中立第三方，KPI：收入和客户满意度 → 天花板：通用数据市场，核心场景被客户自建后空间压缩 │ ▼ 底层采集基础设施：数采中心 / 劳务中介 / 一线采集员
📡 判断数据团队战略地位的信号
被重视的信号
  • 数据 PM 参与模型训练策略讨论（不只是执行）
  • 有专属的数据工程基础设施预算
  • 数据质量问题可以 Block 模型发布
边缘化的信号
  • 只做执行，研究员说什么采什么
  • 没有自主的供应商管理权限
  • 数据质量问题经常被"先用着再说"
长期价值判断

Pre-train/Mid-train 数据的供应链管理能力，在大模型行业里是真正的稀缺能力。合规体系（版权授权）、质量体系（五层验证）、供应商体系（SOW + 管理），三块能同时建好的人极少。

随着模型能力逼近公开数据天花板，高质量过程性数据的稀缺程度只会越来越高——公开互联网数据已经被反复训练，边际价值在下降；但"专家真实推理过程"从未被大规模系统采集过，这个缺口不会自动填上。

做好这个岗位的核心：不是"采了多少数据"，而是"采的数据有多少真正转化成了模型能力提升"。建立可量化的数据价值评估体系，让每一次采购决策都有数据支撑，才是这个岗位真正的竞争力。
